# Threat Model

Last updated: 2026-04-08

## Trust Boundaries

```
Operator (human)
  |
  |-- Telegram channel ──────────┐
  |-- Dashboard web UI ──────────┤
  |                              v
  |                     Runtime Kernel (daemon.ts)
  |                       |  Orchestrator, planner, approval bridge
  |                       |
  |                       v
  |                     Worker Pool (in-process / child-process)
  |                       |  17 workers, each with an execution policy
  |                       |
  |                       v
  |                     External Services
  |                       Gmail API, Google Calendar, Google Drive,
  |                       Toggl, Chrome DevTools, LM Studio
  |
  v
SQLite databases (~/.jarvis/)
  runtime.db, crm.db, knowledge.db
```

**Boundary 1 -- Channel ingress to kernel.** Telegram and dashboard are read-only
ingress surfaces. Neither can invoke shell commands or write files directly. All
mutating actions flow through the job queue and approval pipeline.

**Boundary 2 -- Kernel to workers.** The orchestrator dispatches job envelopes to
workers. Workers execute within execution policies (timeout, approval guard).
Credential scoping (`getCredentialsForWorker`) limits which secrets each worker can
access.

**Boundary 3 -- Workers to external services.** Workers call external APIs using
scoped credentials. OAuth tokens are held in `~/.jarvis/config.json` and only
distributed to workers that need them.

## Credential Exposure Surface

| Worker | Credentials received | Justification |
|---|---|---|
| email | gmail OAuth | Reads/sends email via Gmail API |
| calendar | calendar OAuth | Reads/writes Google Calendar |
| browser | chrome debugging URL | Controls Chrome via DevTools protocol |
| social | chrome debugging URL | Uses browser adapter for social platforms |
| time | toggl API token | Reads/writes Toggl time entries |
| drive | drive OAuth | Reads/writes Google Drive files |
| inference | none (uses lmstudio_url from runtime) | Local inference server, no secret needed |
| document, crm, web, office, interpreter, voice, security, system, device, files | none | No external credentials required |

The `anthropic_api_key`, `webhook_secret`, and `telegram` credentials are never
passed to workers. They are consumed only by the kernel (daemon) and dashboard
server process.

## Attack Vectors Addressed

**Prompt injection via chat channels.** Telegram and dashboard chat are ingress
only. Agent plans are generated by the planner, not executed verbatim from user
input. Mutating actions (email.send, social.post, trade_execute) require explicit
approval before execution.

**Approval bypass.** Defense-in-depth: the orchestrator enforces approval checks
before dispatching jobs, and the worker registry has a secondary approval guard
that rejects unapproved envelopes for critical job types. Of 143 job types, 17
always require approval and 33 are conditionally gated.

**Action classification.** The action classifier defaults unknown action suffixes
to mutating (not read-only). This safe default means new job types require
approval until explicitly classified as read-only.

**Filesystem escape.** The files worker operates under a filesystem policy with
explicit allowed roots (`~/.jarvis`, `os.tmpdir()`, project root). System
directories (`/etc`, `C:\Windows`, `.ssh`, `.aws`, `.gcp`, `.azure`), browser
profiles, and credential file patterns (`.env`, `.pem`, `.key`, `id_rsa`) are
blocked by hardcoded deny lists.

**Credential scoping.** `getCredentialsForWorker()` returns only the subset of
config each worker needs. Workers not in the scope map receive an empty credential
object.

**Dashboard auth.** Bearer token authentication with role-based access control
(admin/operator/viewer). Appliance mode enforces that a token must be configured
and warns if bind host is `0.0.0.0`.

**CORS restriction.** Dashboard API restricts `Access-Control-Allow-Origin` to
the configured origin (defaults to `localhost`).

## Attack Vectors NOT YET Addressed

**Worker process isolation is cooperative, not enforced.** Most workers run
`in_process` (same Node.js process as the daemon). A compromised worker can read
the full process memory, including all credentials. The `child_process` isolation
mode for some workers (browser, interpreter, files, device, voice, security,
social) is a policy label but does not enforce an OS-level sandbox.

**No TLS on local APIs.** The dashboard HTTP server and LM Studio inference
endpoint communicate over plaintext HTTP on localhost. An attacker with local
network access could intercept traffic. This is acceptable for single-machine
deployments but not for remote access.

**No credential encryption at rest.** OAuth tokens, API keys, and the Telegram
bot token are stored as plaintext JSON in `~/.jarvis/config.json`. File
permissions are the only protection.

**No rate limiting on the dashboard API.** The Express server does not enforce
request rate limits. A local attacker could flood the API.

**Telegram bot token is a single shared secret.** Anyone with the bot token and
chat ID can send messages to the Telegram channel. There is no per-user
authentication on the Telegram surface.

**No integrity verification on SQLite databases.** The runtime, CRM, and
knowledge databases have no checksums or tamper detection. A local attacker could
modify database contents directly.

**Inference prompt injection.** Agent system prompts are static, but job inputs
can contain adversarial content. There is no output filtering or sandboxed
evaluation of LLM responses before they become plan steps.

**No audit log for credential access.** When a worker accesses credentials, this
is not logged. There is no way to retrospectively determine which workers accessed
which secrets during a run.

## Security Invariants

1. **No mutating action executes without approval when required.** The approval
   pipeline is enforced at two layers (orchestrator + worker registry guard).

2. **Read-only actions never require approval.** Actions classified by suffix
   (search, list, get, scan, etc.) bypass the approval pipeline.

3. **Unknown actions default to mutating.** New job types are treated as requiring
   approval until explicitly classified.

4. **Workers receive only scoped credentials.** The `getCredentialsForWorker`
   function is the single point of credential distribution.

5. **Filesystem access is deny-by-default.** The files worker only accesses paths
   within explicitly allowed roots, and system/credential paths are hardcoded as
   blocked.

6. **Dashboard binds to localhost by default.** Remote access requires explicit
   `JARVIS_BIND_HOST` configuration, and appliance mode warns on `0.0.0.0`.

7. **Channel ingress is read-only.** Telegram and dashboard chat cannot directly
   trigger shell execution, file writes, or email sends without flowing through
   the job queue.
